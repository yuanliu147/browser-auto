# 中后台浏览器自动化编排器 — 问题清单

> 围绕"中后台场景的可视化编排 + 缓存复用"这一核心,需要解决的上层问题清单。
> 底层执行能力(CDP/Playwright)假定已具备,清单聚焦"编排器"自身要解决的难题。

---

## A. 节点抽取与定位 (4 项)

### A1. 节点粒度

**问题**:抽取出来的节点是「按钮 / 输入框」级别,还是「Ant Design Table 一整行」级别?
**关键点**:粒度太细 → 编排冗长;粒度太粗 → 灵活性丢失。需要支持"组合节点"概念。
**建议**:默认按组件级(`.ant-table-row`、`.ant-form-item`)抽取,允许用户下钻到原子元素。

### A2. 稳定描述符

**问题**:同一个按钮,刷新页面后 className 变了、DOM 顺序变了,如何保证下次仍能找到?
**关键点**:不能依赖 `nth-child` 这类位置选择器。
**建议**:多源描述符融合 — `data-testid` > `aria-label` > 文本内容 > 组件类型+父级语义路径,形成 fallback 链。

### A3. 懒加载 / 虚拟列表

**问题**:Ant Design Table 虚拟滚动、Tree 懒展开、Tab 切换才挂载 — 节点抽取时它们不在 DOM 里。
**关键点**:静态扫描会漏。
**建议**:抽取阶段触发"探测式交互"(滚动到底、点开所有 Tab、展开 Tree 一层),记录"该节点需要先做 X 才能出现"。

### A4. 多候选选择器

**问题**:同一个语义节点(如"提交按钮")可能有多个 DOM 候选,编排时如何选?
**关键点**:用户编排时不应该看到 5 个一模一样的"提交"。
**建议**:抽取层做去重合并,运行时再做歧义消解(高亮所有候选让用户选)。

---

## B. 流程编排 (4 项)

### B1. 流程拓扑

**问题**:支持线性、DAG(并行分支)、还是 Tree(条件分支)?
**关键点**:三种复杂度差距巨大,直接做 DAG 会让 v0.1 永远做不完。
**建议**:v0.1 只做线性 + 条件分支 + 循环;DAG 留到 v1.0。

### B2. 循环节点

**问题**:"对表格每一行执行 X"如何表达?循环变量如何在子节点里引用?
**关键点**:中后台高频需求(批量审批、批量导出)。
**建议**:循环节点显式声明输入数组 + 当前项变量名 `${item.name}`;支持 break/continue。

### B3. 条件节点

**问题**:"如果某字段为空则跳过"、"如果弹窗出现则关闭"如何表达?
**关键点**:条件判断的"事实来源"是 DOM 状态、提取数据、还是上一步返回值?
**建议**:条件表达式支持三类来源:`$dom.exists("...")`、`$extract.field`、`$prev.result`。

### B4. 跨步骤数据流

**问题**:第 1 步提取的表格数据,第 5 步要用 — 数据如何跨节点传递?
**关键点**:不能让用户写代码绑变量。
**建议**:每个节点输出 schema 化数据,编排器自动呈现"上文变量"下拉框,可视化连线。

---

## C. 状态识别 (3 项)

### C1. 操作完成判定

**问题**:点了"保存"按钮,什么时候算"操作成功"?Toast 出现?Loading 消失?网络请求完成?URL 变了?
**关键点**:不同业务标准不同,纯靠 LLM 判断慢且贵。
**建议**:节点上挂"完成断言"(默认 Loading 消失 + 无 error toast;高级用户可自定义),失败时再 fallback 到 LLM。

### C2. 弹窗 / 引导层

**问题**:首次访问触发新手引导 mask、不定时出现的运营弹窗,会卡死流程。
**关键点**:中后台系统极常见。
**建议**:全局"干扰拦截器"机制 — 用户登记一组"已知干扰"的关闭选择器,执行前自动关闭。

### C3. 登录失效

**问题**:跑到一半 cookie 过期跳到登录页,流程断了。
**关键点**:中后台 SSO 多,每家不一样。
**建议**:节点级"登录态守卫" — 检测到登录页自动暂停,触发"重登录子流程"(用户预先编排好),完成后回到原位继续。

---

## D. 错误恢复 (3 项)

### D1. 选择器失效降级链

**问题**:首选选择器找不到 → 是直接报错,还是尝试备选?
**关键点**:稳定性的核心。
**建议**:运行时按 A2 的 fallback 链依次尝试,全部失败才报错;并把"哪一级 fallback 命中了"上报到观测层(用于发现 UI 改版)。

### D2. 重试 / 退避

**问题**:网络抖动 / 后端 500,重试几次?间隔多少?
**关键点**:幂等性问题 — 提交订单重试两次会重复下单。
**建议**:节点显式声明"是否幂等",非幂等节点失败必须人工确认才重试;指数退避是默认。

### D3. 部分失败回滚

**问题**:批量操作 100 行,第 30 行失败 — 前 29 行已经改了,怎么办?
**关键点**:浏览器自动化没有事务概念。
**建议**:不强求回滚(做不到),但要保证"断点可续" — 记录已完成项,失败重跑时从断点开始。

---

## E. 业务语义层 (3 项)

### E1. Ant Design / Element / Arco 抽象

**问题**:每次都让用户去点"表格第三行的省略号菜单的编辑按钮",太繁琐。
**关键点**:这是相对 Stagehand 的核心差异点。
**建议**:内置组件适配器:`page.antTable.row({name: '张三'}).action('编辑')`,把 3 步 DOM 操作折叠成 1 步语义。

### E2. 业务断言

**问题**:"导出 CSV 后校验文件不为空"、"提交后表单字段值正确" — 这种业务级校验如何写?
**关键点**:不能让用户写代码。
**建议**:可视化断言节点 — 选目标节点 + 选断言类型(包含/等于/非空)+ 选值来源。

### E3. 国际化

**问题**:同一个系统,中文版按钮叫"编辑",英文版叫"Edit",流程跑不通。
**关键点**:跨国公司中后台高频问题。
**建议**:文本类选择器自动多语言映射(用户登记一次,编排时仍按中文,运行时自动翻译)。

---

## F. 可观测 / 协作 (3 项)

### F1. Trace 采集

**问题**:流程跑失败了,如何让用户(非技术人员)看懂哪一步出问题?
**关键点**:不是给开发者看的 Playwright trace.zip,是给业务人员看的"剧本回放"。
**建议**:每节点录制截图前后对比 + 关键 DOM 状态 + 提取数据,用类似 Loom 的形式呈现。

### F2. 选择器健康度 KPI

**问题**:某个流程跑了 100 次,有几次因为选择器失效降级了?
**关键点**:UI 改版的预警信号。
**建议**:面板呈现"选择器命中率"、"fallback 触发率"、"LLM 兜底次数",降级率超阈值告警。

### F3. 流程分享 / 复用

**问题**:A 编了一个"批量审批"流程,B 想用 — 怎么分享?B 的环境登录态、租户参数都不一样。
**关键点**:中后台流程价值在沉淀。
**建议**:流程导出为 YAML(参考 Midscene),变量与凭证显式声明 + 占位,B 导入后填变量即可跑。

## 附:动作策略 (Action Policies)

> Stagehand 没做、agent-browser 把它当卖点、Playwright 完全可以实现

### 1. 域名白名单

- **场景**:防止 LLM 被恶意页面诱导跳到钓鱼站
- **Playwright 实现**:`page.route('**/*', route => 检查 url.host 是否在白名单)`

### 2. 动作过滤

- **场景**:禁止下载、禁止打开新 Tab、禁止访问 file:///
- **Playwright 实现**:封装 `download()`、`page.on('popup')`、`waitForEvent('download')` 加策略检查

### 3. 数据脱敏

- **场景**:截图 / Trace 里身份证号、手机号要打码
- **Playwright 实现**:`addInitScript` 注入 DOM 改写;截图前在 mask 元素上加遮罩层

### 4. 审计日志

- **场景**:合规要求每次自动化操作可追溯
- **Playwright 实现**:每个节点执行前后写入结构化日志(谁、何时、对什么页面、做了什么、结果)

## 附:Daemon 持续持有浏览器

> agent-browser 用 Rust 做后台进程长期持有 Chrome,CLI 调用通过 IPC

### 优势

- **冷启动消除**:每次执行不必新开 Chrome(节省 1~3 秒)
- **登录态零成本**:浏览器一直开着,cookie / session 不丢
- **跨流程共享上下文**:同一窗口连续跑多个流程,登录一次到底

### Playwright 是否可以?

**完全可以**,有 3 种实现方式:

#### 方式 1:`chromium.launchServer()`

```ts
const server = await chromium.launchServer({ port: 9999 });
// 任意 client: chromium.connect('ws://localhost:9999')
```

#### 方式 2:`connectOverCDP()` 接管已有 Chrome

```bash
chrome --remote-debugging-port=9222 --user-data-dir=/tmp/chrome
```

```ts
const browser = await chromium.connectOverCDP("http://localhost:9222");
```

#### 方式 3:自建 daemon HTTP 服务包一层

- Node.js 进程持有 Playwright `browser` 实例
- HTTP / IPC API 接收 CLI 调用,在已有 page 上执行
- 等价于自己写一个 mini-Browserbase

### 结论

agent-browser 选择 Rust + 原生 CDP 是因为:

1. 二进制分发(不需要 Node.js 环境)
2. Vercel 商业战略(对接自家 Browserbase 云)

这两点对 **个人开发者 + 开源工具 + 中后台场景** 都不成立 → 用 Playwright 即可,daemon 模式当成「v0.5 加分项」打包成产品力。

## 一句话总结

把 **"动作策略 + Daemon 持有 + 中后台组件适配 + 可视化编排"** 四件事打包成一个产品,你的卖点不是「另一个 Stagehand」,而是 **「面向中后台合规自动化的可视化编排器」**。
