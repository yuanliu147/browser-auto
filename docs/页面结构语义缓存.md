# 页面结构语义缓存（PSG）设计文档

## 1. 核心定位

PSG（Page Structure Graph）不是"步骤缓存"，而是**页面认知缓存**。

- **步骤缓存**记住的是"怎么做"（点击哪里、输入什么）
- **PSG**记住的是"这是什么"（这个页面是列表页，表格有 8 列，第 3 列是金额）

**核心价值**：让 Agent 第二次面对同类页面时，不需要重新探索页面结构，直接"查字典"定位字段。

### 2.4 角色/权限隔离（RBAC）

同一 URL，不同角色看到的页面结构可能完全不同。

**问题场景：**

- Admin 访问 `/users`：看到 8 列 + 删除/编辑按钮
- 运营访问 `/users`：看到 5 列 + 仅查看按钮
- 如果 Admin 先访问，PSG 缓存了 8 列，运营复用时找不到自己可见的字段

**解法：PSG 按角色隔离**

```json
{
  "page_instances": {
    "/users": {
      "role_context": "admin",
      "columns": [
        "id",
        "name",
        "email",
        "phone",
        "role",
        "status",
        "created_at",
        "actions"
      ],
      "visible_buttons": ["edit", "delete", "export"]
    },
    "/users": {
      "role_context": "operator",
      "columns": ["id", "name", "email", "status"],
      "visible_buttons": ["view"]
    }
  }
}
```

**角色检测方式：**

```javascript
async detectRole(page) {
  // 1. 从 Cookie/LocalStorage 读取角色标识
  const roleFromStorage = await page.evaluate(() => {
    return localStorage.getItem('userRole') || sessionStorage.getItem('role');
  });

  // 2. 从页面右上角用户名区域推断
  const userText = await page.locator('.user-info, .avatar-dropdown').innerText();
  if (userText.includes('管理员')) return 'admin';
  if (userText.includes('运营')) return 'operator';

  // 3. 从可见菜单项推断
  const menuItems = await page.locator('.sidebar-menu-item').allInnerTexts();
  if (menuItems.includes('系统设置')) return 'admin';

  return roleFromStorage || 'default';
}
```

**查询时的角色匹配：**

```javascript
async getPSG(siteId, path, role) {
  // 优先匹配精确角色
  let psg = await this.store.find({ siteId, path, role });

  // 精确角色不存在，尝试降级到 default
  if (!psg && role !== 'default') {
    psg = await this.store.find({ siteId, path, role: 'default' });
  }

  // 仍不存在，返回 null 触发首次探索
  return psg;
}
```

## 3. 分层匹配策略

新页面打开时，按优先级逐级匹配：

| 层级 | 策略               | 适用场景                             | 耗时     |
| ---- | ------------------ | ------------------------------------ | -------- |
| L1   | **同站匹配**       | 访问过该站点的其他页面               | < 10ms   |
| L2   | **框架识别**       | Ant Design / Element UI / Vuetify 等 | < 100ms  |
| L3   | **页面类型匹配**   | 识别为"列表页/表单页/详情页"         | < 100ms  |
| L4   | **Agent 首次探索** | 完全陌生的系统                       | 3-5 分钟 |

### 3.3 SPA 路由边界判定

React/Vue 应用切换路由时不刷新页面，`history.push('/orders')` 后 URL 变了但 DOM 是增量更新。

**陷阱：**

- 按 URL 判定"新页面"可能误判
- 按 DOM 变化判定，弹窗、Tab 切换也会被误判

**判定逻辑：**

```javascript
class PageBoundaryDetector {
  async detectNavigation(page, oldUrl, newUrl) {
    const oldPath = new URL(oldUrl).pathname;
    const newPath = new URL(newUrl).pathname;

    // 1. URL path 变化 → 可能是新页面
    if (oldPath !== newPath) {
      const layoutChanged = await this.hasLayoutChanged(page);
      return layoutChanged ? "NEW_PAGE" : "SPA_ROUTE_CHANGE";
    }

    // 2. 同一 URL，query 参数导致内容完全不同
    const oldType = new URL(oldUrl).searchParams.get("type");
    const newType = new URL(newUrl).searchParams.get("type");
    if (oldType && newType && oldType !== newType) {
      return "CONTENT_SWITCH";
    }

    return "SAME_PAGE";
  }

  async hasLayoutChanged(page) {
    // 检测核心 layout 模块是否变化
    const sidebar = await page.locator(".sidebar, .sider").count();
    const header = await page.locator(".header, .navbar").count();
    // layout 稳定 = 可能只是内容区切换
    return { layout_stable: sidebar > 0 && header > 0 };
  }
}
```

**策略：**

- `NEW_PAGE` → 重建 PSG
- `SPA_ROUTE_CHANGE` → 保留 layout 级 PSG，重建 content 级实例
- `CONTENT_SWITCH` → 同一实例内记录不同 query 的状态
- `SAME_PAGE` → 弹窗/Tab 切换，不触发重建

### 4.2 敏感数据脱敏

PSG 缓存的 `sample_value` 可能是真实用户数据（手机号、订单号、金额），存在隐私泄露风险。

**脱敏规则：**

```javascript
class SensitiveDataMasker {
  mask(value) {
    if (!value || typeof value !== "string") return value;

    // 手机号
    if (/^1[3-9]\d{9}$/.test(value)) {
      return value.replace(/(\d{3})\d{4}(\d{4})/, "$1****$2");
    }

    // 身份证号
    if (/^\d{17}[\dXx]$/.test(value)) {
      return value.replace(/(.{6}).{8}(.{4})/, "$1********$2");
    }

    // 银行卡
    if (/^\d{16,19}$/.test(value)) {
      return "**** **** **** " + value.slice(-4);
    }

    // 金额：保留符号和格式，数值匿名化
    if (/^[¥$€]\d+[,.]?\d*/.test(value)) {
      return value.replace(/\d+/, "9999");
    }

    // 订单号/流水号：保留前缀模式
    if (/^[A-Z]{2,6}-\d{6,12}$/.test(value)) {
      return value.replace(/\d{6,12}/, "********");
    }

    // 邮箱
    if (/^[^@\s]+@[^@\s]+\.[^@\s]+$/.test(value)) {
      const [name, domain] = value.split("@");
      return name[0] + "***@" + domain;
    }

    return value;
  }
}
```

**存储时自动脱敏：**

```javascript
async extractColumnSample(cell) {
  const rawValue = await cell.innerText();
  return {
    sample_value: masker.mask(rawValue),      // 脱敏后存储
    sample_pattern: inferPattern(rawValue),   // 保留模式：如 "ORD-YYYYMMDD-NNN"
    is_sensitive: masker.isSensitive(rawValue) // 标记是否含敏感数据
  };
}
```

**多语言匹配策略：**

```javascript
async findByHeaderText(psg, text, currentLang) {
  // 1. 匹配已知语言样本
  for (const col of psg.columns) {
    const knownTexts = Object.values(col.header_texts || {});
    if (knownTexts.includes(text)) return col;
  }

  // 2. fallback：用 inferred_name 做语义匹配
  for (const col of psg.columns) {
    if (col.inferred_name && text.toLowerCase().includes(col.inferred_name)) {
      return col;
    }
  }

  // 3. 首次遇到新语言：用 LLM 做一次语义推断，结果入缓存
  const inferred = await llm.translateAndMatch(text, psg.columns);
  if (inferred) {
    psg.columns[inferred.index].header_texts[currentLang] = text;
    await this.cache.save(psg);
  }

  return inferred;
}
```

**语言检测：**

async detectLanguage(page) {

## 5. 检测时机：按需验证，非定时巡检

```javascript
async validateBeforeUse(page, pageUrl, adaptation) {
  const instance = adaptation.page_instances[pageUrl.pathname];
  if (!instance) return { action: 'BUILD_COGNITION' };

  const hoursSinceVerify = (Date.now() - new Date(instance.last_verified)) / 3600000;

  if (hoursSinceVerify < 1)   return { action: 'USE_DIRECTLY' };
  if (hoursSinceVerify < 24)  return { action: 'QUICK_CHECK' };      // 验证 1-2 个关键元素
  if (hoursSinceVerify < 168) return { action: 'STANDARD_CHECK' };   // 验证表头和主要操作
  return { action: 'DEEP_REBUILD' };                                  // 深度重建认知
}
```

## 6. 缓存失效与修复策略

### 6.1 失效触发条件

| 类型     | 条件                                   | 检测方式         |
| -------- | -------------------------------------- | ---------------- |
| 硬失效   | 主选择器 + 所有 fallback 连续 3 次失败 | 回放异常捕获     |
| 软失效   | 匹配置信度从 0.9+ 跌至 0.5 以下        | 启发式分数下降   |
| 结构失效 | 父元素选择器失效                       | 祖先节点检测失败 |
| 语义失效 | innerText 相似度 < 30%                 | Levenshtein 距离 |
| 时间失效 | 超过 90 天未使用                       | 定时扫描         |

### 6.2 四级修复响应

| 级别                | 场景                      | 动作                                    | 耗时     |
| ------------------- | ------------------------- | --------------------------------------- | -------- |
| L1 自动修复         | 单元素软失效              | fallback 升级 / 语义重定位 / 视觉重定位 | < 5s     |
| L2 PSG 局部修复     | 表格列偏移 / 模块位置调整 | 更新列索引 / 模块选择器                 | < 10s    |
| L3 Agent 局部重探索 | PSG 部分失效              | 限定范围重新探索失效步骤                | 30s-2min |
| L4 人工修复         | 彻底改版                  | 通知用户，截图标注，等待点选            | 人工时间 |

### 6.3 缓存生命周期状态机

```
ACTIVE → (soft_fail) → DEGRADED → (auto_repair_fail) → REPAIRING → (repair_success) → ACTIVE
                                                            ↓
                                                     (repair_fail) → PENDING_HUMAN → (human_fix) → ACTIVE
                                                                                ↓
                                                                          (timeout) → DEPRECATED
```

## 7. 页面大升级应对

### 7.1 自动检测升级

```javascript
async detectChange(page, workflow) {
  const sampleSteps = workflow.steps.slice(0, 3);
  let matchCount = 0;
  for (const step of sampleSteps) {
    const cached = this.cache.get(step.id);
    try { await this.matcher.findElement(page, cached); matchCount++; } catch {}
  }
  const matchRate = matchCount / sampleSteps.length;

  if (matchRate < 0.5) return { status: 'MAJOR_UPGRADE', message: '页面可能已大改版' };
  if (matchRate < 1)   return { status: 'MINOR_CHANGE', message: '部分元素已变化' };
  return { status: 'OK' };
}
```

## 8. 多系统场景的复用策略

### 8.1 复用层级

| 层级                 | 覆盖场景                   | 预计占比 |
| -------------------- | -------------------------- | -------- |
| 同站 adaptation 复用 | 一个系统内多个页面         | ~30%     |
| 同框架模板命中       | Ant Design / Element UI 等 | ~40%     |
| 页面类型指导探索     | 老系统、自定义系统         | ~20%     |
| 完全未知需首次探索   | 极其特殊的系统             | ~10%     |

### 8.2 异类系统降级

对 jsp/php 老系统、高度定制系统，降级到**纯视觉 + 语义匹配**：

- 不依赖 table 标签，找"长得像表格"的区域（等宽横向排列的子元素）
- 按钮是图片时，用视觉哈希匹配
- 纯文本分页用 OCR/文本模式识别

### 8.3 冷启动优化

完全陌生的系统，Agent 首次探索需要 3-5 分钟，用户体验差。

**优化策略 A：利用公开文档预加载**

```javascript
async preloadFromDocs(siteUrl) {
  // 尝试抓取帮助文档、API 文档
  const possibleDocs = [
    `${siteUrl}/help`,
    `${siteUrl}/docs`,
    `${siteUrl}/manual`,
    `${siteUrl}/api/docs`
  ];

  for (const docUrl of possibleDocs) {
    try {
      const docs = await fetch(docUrl);
      const extracted = await llm.extractPageStructures(docs);

      // 预生成半成品 adaptation
      for (const page of extracted) {
        await this.cache.saveDraftAdaptation(siteUrl, page);
      }
      break;
    } catch {
      continue;
    }
  }
}
```

**优化策略 B：快速模式引导（三问法）**

```javascript
async quickPatternBootstrap(page) {
  // 不探索全部细节，只回答三个问题
  const pageType = await this.classifyPageType(page);      // 列表/表单/详情？
  const framework = await this.detectFramework(page);      // 什么框架？
  const contentArea = await this.locateContentArea(page);  // 核心数据区在哪？

  // 基于这 3 个信息，生成最小可用 PSG
  return {
    pageType,
    framework,
    contentArea,
    completeness: 'partial',
    note: '首次快速扫描，后续逐步完善'
  };
}
```

**优化策略 C：录制屏幕辅助 Debug**

```javascript
async exploreWithRecording(page, task) {
  const videoPath = `./recordings/${Date.now()}.webm`;

  // 启动视频录制
  await page.video()?.saveAs(videoPath);

  try {
    const result = await this.agent.explore(page, task);
    return result;
  } catch (err) {
    // 探索失败时，用户可回看视频
    console.error(`探索失败，录制已保存至 ${videoPath}`);
    throw err;
  }
}
```

**优化策略 D：同框架模板秒级加载**

```javascript
async coldStartWithTemplate(page, siteUrl) {
  const framework = await this.detectFramework(page);

  if (framework !== 'unknown') {
    // 直接加载预置的框架模板，跳过探索阶段
    const template = await this.loadFrameworkTemplate(framework);
    const adaptation = await this.instantiateTemplate(template, siteUrl);

    console.log(`检测到 ${framework}，已加载框架模板，跳过基础探索`);
    return adaptation;
  }

  // 未知框架，走完整探索
  return await this.agent.explore(page);
}
```

## 9. 存储量估算

| 层级                   | 数量               | 单条大小 | 总大小     |
| ---------------------- | ------------------ | -------- | ---------- |
| 页面模式（Pattern）    | 30 个（通用）      | 10 KB    | 300 KB     |
| 站点适配（Adaptation） | 100 站点 × 10 模式 | 20 KB    | 20 MB      |
| 页面实例（Instance）   | 1000 个活跃 URL    | 5 KB     | 5 MB       |
| **总计**               |                    |          | **~25 MB** |

对比按 URL 存 PSG：10000 URL × 15 KB = **150 MB**，且大量重复。

## 10. 核心设计原则

1. **按模式聚类，不按 URL 存储**：网页再多，模式有限（列表页、表单页、详情页、Dashboard）
2. **Agent 首次探索构建认知，后续查字典复用**：PSG 是加速器，不是替代器
3. **按需验证，非定时巡检**：要用的时候现场验证，不用的时候不浪费资源
4. **分层匹配 + 多维度 fallback**：同站 → 同框架 → 页面类型 → 视觉语义，逐级降级
5. **失效不是删除，是分级修复**：自动修复 → 局部更新 → 人工修复，最大化复用已有认知
